# Подробное описание функционала проекта

## Общая идея

Проект представляет собой Telegram-бота для анонимной предложки.

Пользователь пишет боту в личные сообщения. Бот:

1. принимает сообщение;
2. проверяет его через локальный фильтр запрещённого контента;
3. если сообщение чистое, отправляет его в чат модерации;
4. модератор принимает решение;
5. при одобрении бот публикует сообщение в целевой чат или канал;
6. при отклонении бот уведомляет автора.

Главная особенность проекта в том, что он не использует AI, агентов или внешние сервисы анализа текста. Вся фильтрация работает локально по правилам.

## Основные сценарии работы

### 1. Пользователь отправляет сообщение

Пользователь открывает личный чат с ботом и отправляет текст.

Сейчас бот принимает:

- обычные текстовые сообщения;
- фото с подписью.

Сообщения из групп и каналов как новые заявки не принимаются.

### 2. Бот фильтрует сообщение

Перед отправкой на модерацию бот прогоняет текст через фильтр.

Фильтр проверяет:

- запрещённые слова и выражения;
- ссылки и внешние контакты;
- слишком длинные сообщения.

Если фильтр находит проблему, бот не отправляет сообщение в предложку и отвечает пользователю, что в тексте есть запрещённые слова, выражения или ссылки.

### 3. Бот отправляет заявку в чат модерации

Если сообщение прошло фильтр, бот создаёт запись в базе данных и отправляет заявку в чат модерации.

В сообщении для админов указываются:

- внутренний ID заявки;
- тип заявки: текст или фото;
- username отправителя или имя, если username нет;
- текущий статус;
- кто принял решение;
- время решения;
- причина отклонения, если заявка отклонена;
- сам текст заявки;
- фото, если оно было приложено автором.

Пока заявка не обработана, внизу сообщения показываются inline-кнопки:

- `Опубликовать`
- `Отклонить`

### 4. Модератор принимает решение

Когда админ нажимает кнопку:

- при `Опубликовать` бот отправляет текст в целевой чат;
- при `Отклонить` бот меняет статус на отклонённый.

После этого бот:

- обновляет запись в SQLite;
- записывает, кто именно принял решение;
- записывает время решения;
- обновляет сообщение в чате модерации;
- уведомляет автора заявки о результате.

## Фильтрация запрещённого контента

### Источник словаря

Список запрещённых слов и выражений хранится в файле:

- `data/banned-terms.txt`

Файл можно редактировать вручную. Слова и фразы перечисляются через запятую. Можно переносить список на новые строки.

Пример:

```txt
секс, порно, идиот, нацизм, white power
```

### Что умеет фильтр

Фильтр умеет находить:

- обычные запрещённые слова;
- слова с подменой похожих букв латиницей и кириллицей;
- слова, разбитые пробелами;
- слова, разбитые символами вроде `.`, `-`, `*`, `_`, `/`, `\`.

Примеры того, что должно блокироваться:

- `секс`
- `сekс`
- `с е к с`
- `c.e.k.c`
- `п*о*р*н*о`
- `н-а-ц-и-з-м`

### Ограничения текущего словаря

Сейчас словарь максимально упрощён: это просто список слов и фраз.

Это удобно для ручного редактирования, но есть особенность:

- формы слов не выводятся автоматически.

Например, если в списке есть `идиот`, но нет `идиоты`, то для полной жёсткости иногда стоит явно добавить и вторую форму.

## Хранение данных

### Где хранится история

История хранится в SQLite-файле:

- `data/submissions.db`

Это обычная файловая база данных. Отдельный сервер базы данных не нужен.

### Что хранится в базе

Для каждой заявки сохраняются:

- уникальный ID заявки;
- ID пользователя Telegram;
- username;
- имя;
- текст сообщения;
- дата создания;
- статус заявки;
- ID сообщения в чате модерации;
- ID опубликованного сообщения;
- причина отклонения;
- ID модератора;
- username модератора;
- имя модератора;
- время принятия решения.

### Почему SQLite лучше JSON

SQLite выбрана вместо JSON, потому что:

- она надёжнее;
- не нужно читать и перезаписывать весь файл целиком;
- лучше подходит для истории заявок;
- удобнее расширять в будущем;
- меньше риск испортить данные при сбоях.

## Модерация

### Кто может модерировать

Права на модерацию определяются через переменную окружения:

- `ADMIN_USER_IDS`

Если список пустой, то по текущей логике кнопки модерации сможет использовать любой участник чата модерации.

Если список заполнен, кнопки будут работать только для указанных пользователей.

### Как выглядит решение

После модерации заявка получает один из статусов:

- `pending` — ожидает решения;
- `approved` — одобрена;
- `rejected` — отклонена.

## Переменные окружения

Проект использует `.env`.

Основные переменные:

- `BOT_TOKEN` — токен Telegram-бота;
- `MODERATION_CHAT_ID` — чат, куда бот отправляет заявки на модерацию;
- `TARGET_CHAT_ID` — чат или канал, куда бот публикует одобренные сообщения;
- `ADMIN_USER_IDS` — список Telegram user id модераторов через запятую.

## Основные файлы проекта

- `src/index.ts` — основной код Telegram-бота;
- `src/filter.ts` — фильтрация текста;
- `src/storage.ts` — работа с SQLite;
- `src/types.ts` — общие типы;
- `data/banned-terms.txt` — словарь запрещённых слов;
- `data/submissions.db` — история заявок;
- `README.md` — краткая инструкция по запуску.

## Команды проекта

### `npm.cmd run dev`

Режим разработки.

Используется для локальной работы:

- запускает бота из `src`;
- следит за изменениями файлов;
- автоматически перезапускает процесс при изменениях.

### `npm.cmd run build`

Собирает TypeScript из `src` в JavaScript в `dist`.

### `npm.cmd start`

Запускает собранную production-версию из `dist`.
